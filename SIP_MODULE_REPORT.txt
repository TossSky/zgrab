
================================

Что делает модуль
-----------------

Модуль sip отправляет SIP-запросы на указанные цели и разбирает полученные ответы
в структурированный JSON. Это позволяет массово сканировать SIP-серверы (IP-телефонию)
и извлекать из ответов информацию, полезную для анализа безопасности.

Поддерживаемые методы запросов: OPTIONS, REGISTER, INVITE.
Поддерживаемые транспорты: UDP, TCP, TLS (SIPS).

При запуске модуль формирует корректный SIP-запрос по RFC 3261 с заголовками Via,
From, To, Call-ID, CSeq, Contact, User-Agent, Max-Forwards и Content-Length.
Для метода INVITE опционально генерируется SDP-тело с описанием аудиосессии.
Запрос отправляется на целевой хост, ответ принимается, парсится и выдаётся как JSON.


Что извлекается из ответа
-------------------------

Из каждого SIP-ответа модуль извлекает:

- Статусная строка: версия протокола, код ответа, текстовое пояснение.
  Пример: SIP/2.0 200 OK или SIP/2.0 404 Not Found.

- Основные заголовки: From, To, Call-ID, CSeq, Via (включая параметры branch, received,
  rport), Allow, Supported, User-Agent, Server, Contact, Accept, Content-Type,
  Content-Length, WWW-Authenticate.

- Вендорские заголовки: X-Serialnumber и любые другие заголовки, начинающиеся на X- или P-.

- SDP (если присутствует в ответе): IP-адрес соединения, медиапотоки с портами,
  кодеки с параметрами (rtpmap, fmtp), флаги поддержки шифрования (SRTP, DTLS-SRTP, ICE).


Какую информацию по безопасности можно получить
-----------------------------------------------

1. Вендор и версия сервера. Заголовки Server и User-Agent раскрывают тип и версию
   программного обеспечения. Это позволяет искать известные CVE для конкретных версий.
   Например, Asterisk PBX 1.6.0.28 или 1.8.11 — сильно устаревшие версии с множеством
   известных уязвимостей.

2. Разрешённые методы. Заголовок Allow показывает, какие SIP-методы принимает сервер.
   Наличие REGISTER означает, что сервер принимает регистрации (потенциально можно
   подобрать учётные данные). Наличие INVITE означает, что сервер обрабатывает звонки.

3. Аутентификация. При отправке REGISTER сервер может вернуть 401 Unauthorized
   с заголовком WWW-Authenticate, из которого видно тип аутентификации (Digest),
   realm и используемый алгоритм.

4. Шифрование медиа. Если в SDP присутствует протокол RTP/SAVP или RTP/SAVPF,
   сервер поддерживает SRTP. Наличие атрибутов fingerprint и setup указывает на
   поддержку DTLS-SRTP. Атрибуты ice-ufrag и ice-pwd указывают на поддержку ICE.
   Отсутствие всего этого означает, что голосовой трафик передаётся без шифрования.

5. Сетевая информация. Заголовок Via в ответе содержит параметры received и rport,
   из которых виден реальный IP-адрес и порт сканера с точки зрения сервера.
   Заголовок Contact содержит адрес, на который сервер ожидает дальнейшие запросы.

6. Серийные номера устройств. Некоторые SIP-устройства (IP-телефоны, шлюзы)
   отправляют заголовок X-Serialnumber, по которому можно идентифицировать
   конкретное оборудование.


Флаги командной строки
----------------------

--port            Порт назначения (по умолчанию 5060).
--udp             Использовать UDP транспорт.
--tls             Использовать TLS (SIPS, подразумевает TCP).
--method          SIP-метод: OPTIONS, REGISTER или INVITE (по умолчанию OPTIONS).
--from            SIP From URI. Генерируется автоматически, если не указан.
--to              SIP To URI. Генерируется автоматически, если не указан.
--contact         SIP Contact URI. Генерируется автоматически, если не указан.
--user            Имя пользователя для SIP URI (по умолчанию probe).
--domain          Домен для SIP URI. Используется IP цели, если не указан.
--no-sdp          Не включать SDP-тело в INVITE-запросы.
--read-timeout    Таймаут чтения ответа в миллисекундах (по умолчанию 5000).
--max-tries       Количество повторных попыток для UDP (по умолчанию 3).
--user-agent      Значение заголовка User-Agent (по умолчанию zgrab2/sip).
--timeout         Общий таймаут соединения (по умолчанию 10s).


Примеры запуска
---------------

OPTIONS по UDP:
  echo "192.168.1.1" | ./zgrab2.exe sip --port 5060 --method OPTIONS --udp

OPTIONS по TCP:
  echo "192.168.1.1" | ./zgrab2.exe sip --port 5060 --method OPTIONS

REGISTER по TLS:
  echo "192.168.1.1" | ./zgrab2.exe sip --port 5061 --method REGISTER --tls

INVITE с SDP:
  echo "192.168.1.1" | ./zgrab2.exe sip --port 5060 --method INVITE --udp --user alice

Массовое сканирование из файла с сохранением результата:
  cat targets.txt | ./zgrab2.exe sip --port 5060 --method OPTIONS --udp > results.json 2>scan.log


Куда сохраняется результат
--------------------------

По умолчанию результат выводится в стандартный вывод (stdout). Каждая строка вывода
представляет собой отдельный JSON-объект с результатом сканирования одного хоста.
Служебные сообщения (время старта, финиша, статистика) выводятся в stderr.

Для сохранения в файл нужно перенаправить stdout:
  cat targets.txt | ./zgrab2.exe sip --port 5060 --method OPTIONS --udp > results.json

В файле results.json каждая строка будет содержать JSON с полями ip, data.sip.status,
data.sip.protocol, data.sip.result (ответ сервера с заголовками и SDP).


Структура файлов модуля
-----------------------

modules/sip/scanner.go        Основной модуль: флаги, регистрация, метод Scan с UDP/TCP/TLS.
modules/sip/request.go        Генерация SIP-запросов (BuildSIPRequest) и SDP-тела.
modules/sip/parser.go         Парсер SIP-ответов: статусная строка, заголовки, SDP.
modules/sip/parser_test.go    Юнит-тесты парсера (13 тестов, все проходят).
modules/sip.go                Регистрация модуля в zgrab2.
zgrab2_schemas/zgrab2/sip.py  JSON-схема вывода модуля для валидации.
